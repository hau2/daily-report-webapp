  <behavior>
    - getTeamReports(teamId, date): returns array of TeamMemberReport objects (one per team member) with user info, report status, tasks, and totalHours. Members without a report get status 'none' and empty tasks.
    - getPendingSubmissions(teamId, date): returns array of PendingMember objects for members who do NOT have a daily_reports row with status='submitted' for that date. A member with a draft report IS still pending.
    - generateCsv(teamId, date): returns CSV string with header "Member Name,Email,Date,Report Status,Task,Hours,Source Link,Notes". Each task is a separate row. Members with no tasks get one row with empty task fields. Fields containing commas, quotes, or newlines are properly escaped.
    - escapeCsvField: wraps fields with commas/quotes/newlines in double quotes, doubles internal quotes.
  </behavior>
  <action>
    1. Create `packages/shared/src/types/manager.ts` with these types:
       - `TeamMemberReport`: { userId, email, displayName, status: 'draft' | 'submitted' | 'none', report: DailyReport | null, tasks: Task[], totalHours: number }
       - `PendingMember`: { userId, email, displayName, reportStatus: 'draft' | 'none' }
       - `TeamReportsResponse`: { date: string, teamId: string, members: TeamMemberReport[] }
    2. Export from `packages/shared/src/index.ts`
    3. Create test file `apps/api/src/manager/manager.service.spec.ts` with stubs using `createMockSupabaseService()` from `apps/api/test/setup.ts`. Write failing tests FIRST (RED phase):
       - "getTeamReports returns members with their reports and tasks"
       - "getTeamReports returns status 'none' for members without a report"
       - "getPendingSubmissions returns members without submitted reports"
       - "getPendingSubmissions includes members with draft reports as pending"
       - "generateCsv produces correct header and data rows"
       - "generateCsv escapes fields with commas and quotes"
    4. Create `apps/api/src/manager/manager.service.ts` implementing:
       - `getTeamReports(teamId, date)`: Query team_members for userIds, query users for display info, query daily_reports for the date, query tasks for those reports. Assemble per-member view using the same `mapReport`/`mapTask` private helper pattern from TasksService (copy the mappers, don't import -- they're private).
       - `getPendingSubmissions(teamId, date)`: Query team_members, query daily_reports with status='submitted' for that date, diff to find pending. Return user details for pending members. Include draft status vs none distinction.
       - `generateCsv(teamId, date)`: Call getTeamReports internally, flatten into CSV rows. Use `escapeCsvField` helper (private method). Members with no tasks still get a row (status 'none', empty task fields).
    5. Run tests (GREEN phase). All 6 tests must pass.

    IMPORTANT: The mockQueryBuilder from setup.ts uses chaining where all methods return `this` except `single`/`maybeSingle` which resolve. For queries that return arrays (no `.single()`), the final chained method's mock needs to resolve with `{ data: [...], error: null }`. Use `mockQueryBuilder.order.mockResolvedValueOnce(...)` or similar for the terminal call in each chain. Since multiple `from()` calls happen in sequence, use `mockClient.from.mockReturnValueOnce(...)` to return different query builders for different tables, OR reset mocks between calls.

    Recommended approach for testing: Create separate mockQueryBuilder instances per `from()` call using a factory, or use `mockClient.from.mockImplementation((table) => ...)` to return different builders per table name.
